### 느낀점
* 네이밍 항상 생각하자. 명확하게 뭘 보여줄 것인지. 
* 이런식의 리팩토링은 지금이나 가능하다. 파일 좀만 더 많아지면 절대 불가능하다.    
처음부터 꼼꼼하게 하자.
* 메서드 단위로 순수한 단위테스트를 해야 하는데 어떻게 해야 할까   
* Service에서 Controller로 데이터를 넘겨줄 때 DTO를 통해 진행하자.     
처음부터 이렇게 진행되지 않으면 나중에 너무 바꾸기 힘들다. 테스트도 연결되어 있고.    
항상 테스트는 그대로 돌아가는 상태에서 기능 소스를 변경해야 한다.     
* DTO에 생성자 오버로딩을 통해 여러 형태의 데이터를 자유롭게 받아 전달할 수 있도록 하자.     
setter 사용 안 할수도 있을 듯.   
* 테스트 케이스는 항상 그 때 그 때 작성하자. 테스트로 각 메서드를 검증하는 연습을 해야 한다. DB를 통해서가 아니라.   
* Service class는 비즈니스 코드가 위치한다.    
이 메서드들은 하나의 행위만을 해야 한다. 명심하자.    
* Controller는 뷰와 Service의 핸들링 역할이다. Class 존재 이유는 핸들링이다.   
최대한 비즈니스 로직은 빼보자. 뭔가 의존성을 가지고 생성하는 것도 안될 거 같다.
